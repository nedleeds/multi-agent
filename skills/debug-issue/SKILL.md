사용자가 에러·스택트레이스·"X 가 동작 안함"·"NullPointerException" 같은 증상을 보고할 때 로드. 수정부터 뛰어들지 말고 순서대로 진행.

## 1. 재현 & 에러 정확히 읽기
- 정확한 에러 메시지, 스택트레이스, 트리거 명령을 확보.
- 사용자가 스택트레이스를 주지 않았으면 `bash` 로 실패 명령을 돌려 캡처.
- 기록: 실패한 프레임의 `file:line` + 에러 클래스.

## 2. 실패 코드 위치 찾기
- 스택트레이스의 파일/라인을 `read_file` — 앞뒤 ±30줄 함께.
- 해당 함수명을 `grep` 으로 역추적. 호출자 상위 2-3개 읽기.

## 3. 가설을 명시적으로 세움 (한 문장)
"X 가 발생한 이유는 Y 때문이다" 형태로 고정. 가설이 없으면 단계 2로 돌아감.

## 4. 수정하기 전에 가설 검증
다음 중 하나:
- `print(...)` 추가 + `bash` 재실행으로 실제 상태 확인
- `grep` 으로 가설을 증명/반증할 config·상태를 찾기

검증 결과가 가설과 모순하면 단계 3으로 복귀. 검증 없이 수정 금지.

## 5. 최소 수정 적용
- `edit_file` — 가능한 가장 좁은 변경.
- 관련 없는 케이스의 defensive 코드 추가 금지. 리팩터링 끼워넣기 금지.

## 6. 수정 검증
- 원래 실패 명령을 `bash` 로 재실행 → 성공 확인.
- 테스트가 있으면 관련 테스트 실행. 없으면 1줄 smoke 스크립트라도 실행.

## 7. 보고 (3줄)
- **무엇이 잘못됐나** (1줄)
- **왜** (1줄 — 메커니즘)
- **변경** (file:line + 1줄 diff 요약)
